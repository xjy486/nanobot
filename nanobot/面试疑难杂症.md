nanobot 项目中可能被面试官深挖的技术点记录。

## 1. 一次会话最多迭代多少轮？

不同层级的默认上限：

| 层级                    | 配置项                              | 代码位置                                             | 默认值  |
| ----------------------- | ----------------------------------- | ---------------------------------------------------- | ------- |
| **主 AgentRunner**      | `AgentDefaults.max_tool_iterations` | [config/schema.py:80](config/schema.py#L80)          | **200** |
| **子 Agent (subagent)** | 继承主 runner 的 `max_iterations`   | [agent/subagent.py:95-98](agent/subagent.py#L95-L98) | **200** |
| **Dream 记忆巩固**      | `DreamConfig.max_iterations`        | [config/schema.py:48](config/schema.py#L48)          | **15**  |

核心循环在 `agent/runner.py:246`：

```python
for iteration in range(spec.max_iterations):
    # ...每轮：上下文治理 → 调用 LLM → 执行工具 → 检查结果...
else:
    stop_reason = "max_iterations"
    final_content = render_template("agent/max_iterations_message.md", max_iterations=...)
```

- 每次 LLM 返回 tool_calls → 执行工具 → 将工具结果追加到消息列表 → 进入下一轮，这算一次 iteration
- 如果 LLM 直接返回文本（不用工具），本轮就结束，不一定走满 200 轮
- 运行时可通过 `my` 技能动态调整 `max_iterations`，范围 1-100（[agent/tools/self.py:78](agent/tools/self.py#L78)）

子 agent（`Task`/`Agent` 工具）也受同样的 max_iterations 约束，通过 `_sync_subagent_runtime_limits()` 保持同步（[agent/loop.py:311-313](agent/loop.py#L311-L313)）。

## 2. Agent 卡死了怎么办？有什么打断机制？

核心问题：agent 在调用某个工具时反复失败、反复重试同一工具。nanobot 有**五层防线**，但没有专门的"同工具重复调用检测"。

### 第一层：LLM 调用级重试（Provider Retry）

位置：[providers/base.py:96-98](providers/base.py#L96-L98)

```python
_CHAT_RETRY_DELAYS = (1, 2, 4)           # 指数退避：1s → 2s → 4s
_PERSISTENT_MAX_DELAY = 60                # persistent 模式单次最大等待 60s
_PERSISTENT_IDENTICAL_ERROR_LIMIT = 10    # 连续 10 次完全相同的错误内容就放弃
```

两种重试模式（`provider_retry_mode`）：

| 模式               | 行为                                                               | 适用场景             |
| ------------------ | ------------------------------------------------------------------ | -------------------- |
| `standard`（默认） | 最多重试 3 次，1s/2s/4s 退避                                       | 一般使用             |
| `persistent`       | 无限重试，但单次延迟上限 60s，且**同一条错误内容连续 10 次就停止** | 长时间任务、后台运行 |

**重要**：这一层针对的是 **LLM API 调用返回的 transient error**（限流、服务不可用等），不是工具执行失败。

### 第二层：空内容重试

位置：[agent/runner.py:39](agent/runner.py#L39)，`_MAX_EMPTY_RETRIES = 2`

LLM 返回空内容时，额外重试最多 2 次。超过后会尝试 `_request_finalization_retry` 做最后的补救。

### 第三层：长度恢复重试

位置：[agent/runner.py:40](agent/runner.py#L40)，`_MAX_LENGTH_RECOVERIES = 3`

当 `finish_reason == "length"`（输出被截断）时，自动追加 "please continue" 消息让 LLM 继续，最多 3 次。

### 第四层：注入循环限制

位置：[agent/runner.py:41-42](agent/runner.py#L41-L42)

```
_MAX_INJECTIONS_PER_TURN = 3   # 每轮最多注入 3 条用户消息
_MAX_INJECTION_CYCLES = 5      # 最多 5 轮注入循环
```

防止外部注入消息导致无限循环。

### 第五层：max_iterations 硬上限

**最终兜底**：`for iteration in range(spec.max_iterations)` 循环必然在 200 轮后退出，无论中间状态如何。

### 关于"同一工具反复调用失败"的现状

**坦白说，nanobot 目前没有显式的"同一工具连续失败 N 次就打断"的机制。** 现有保护是间接的：

1. 工具执行失败的结果会以 `role: "tool"` 消息返回给 LLM，LLM 看到失败结果后**理论上**会调整策略
2. 如果 LLM 坚持调用同一工具，只能在 200 轮后靠 `max_iterations` 兜底终止
3. provider 级别的"连续相同错误 10 次停止"针对的是 LLM API 响应的文本内容，不是工具调用的语义分析

**改进方向**：可增加 tool execution loop detector——同一工具名 + 相似参数连续失败 ≥ N 次就主动中止并给出明确提示。

## 3. Provider Retry 的完整流程图

```
LLM 调用失败
  ├── 非 transient 错误（如内容过滤、权限拒绝）
  │     ├── 包含图片 → 去掉图片重试一次
  │     └── 否则 → 直接返回错误给用户
  │
  └── transient 错误（限流、503、超时等）
        ├── standard 模式
        │     └── 按 1s/2s/4s 重试，最多 3 次
        │
        └── persistent 模式
              ├── 同一条错误连续出现 10 次 → 放弃
              ├── 单次重试延迟不超过 60s
              └── 理论上无限重试（直到 10 次相同错误或成功）
```

## 5. Session 会话是怎么持久化的？

核心问题：一条消息是生成后立即写入 session，还是等整个循环结束再统一写？

**答案是三段式：用户消息提前写 → 循环中打 checkpoint 快照 → 循环结束后批量写 messages。**

### 第一段：用户消息提前持久化（防崩溃丢输入）

位置：[agent/loop.py:1052-1065](agent/loop.py#L1052-L1065)

```python
user_persisted_early = False
if not pending_ask_id and (has_text or media_paths):
    session.add_message("user", text, **extra)
    self._mark_pending_user_turn(session)
    self.sessions.save(session)        # ← 进入 runner 前就已落盘
    user_persisted_early = True
```

目的是防止循环中途崩溃导致用户输入静默丢失。如果崩溃了，下次恢复时 `_restore_pending_user_turn()` 会检测到 `PENDING_USER_TURN` 标记，自动补一条 "Error: Task interrupted..." 的 assistant 消息，避免出现连续两条 user 消息。

### 第二段：运行中 Checkpoint 快照（写 metadata，不是 messages）

[runner.py](agent/runner.py) 中 `_emit_checkpoint()` 在三个关键节点写入 `session.metadata`：

| 时机                            | phase               | 代码位置                                       | 存了什么                                   |
| ------------------------------- | ------------------- | ---------------------------------------------- | ------------------------------------------ |
| LLM 返回 tool_calls，工具执行前 | `"awaiting_tools"`  | [runner.py:298-308](agent/runner.py#L298-L308) | assistant_message + pending_tool_calls     |
| 工具执行完成后                  | `"tools_completed"` | [runner.py:363-373](agent/runner.py#L363-L373) | assistant_message + completed_tool_results |
| LLM 返回最终文本后              | `"final_response"`  | [runner.py:512-522](agent/runner.py#L512-L522) | 最终的 assistant_message                   |

checkpoint 写入路径（[loop.py:1232-1235](agent/loop.py#L1232-L1235)）：

```python
def _set_runtime_checkpoint(self, session, payload):
    session.metadata[self._RUNTIME_CHECKPOINT_KEY] = payload
    self.sessions.save(session)     # ← checkpoint 数据每次立即写盘
```

注意：checkpoint 写的是 `session.metadata`，不是 `session.messages`。它是一个**运行态快照**，用于 `/stop` 中断后恢复部分上下文。

**恢复机制**（[loop.py:1259-1311](agent/loop.py#L1259-L1311)）：下次进入 `_process_message` 时，`_restore_runtime_checkpoint()` 检查 metadata 中是否有未完成的 checkpoint，如果有就物化到 `session.messages` 中：

```
checkpoint 中的 assistant_message → append 到 messages
checkpoint 中的 completed_tool_results → append 到 messages
checkpoint 中的 pending_tool_calls → 为每个未完成的 tool_call 补一条 error 占位消息
```

恢复时会做**去重**：用 `_checkpoint_message_key()` 提取消息的关键字段（role、content、tool_call_id、name、tool_calls 等），与现有 messages 尾部比对，跳过已存在的重叠部分（[loop.py:1296-1306](agent/loop.py#L1296-L1306)）。

### 第三段：循环结束后批量写入 messages

位置：[loop.py:1067-1091](agent/loop.py#L1067-L1091)

```python
final_content, _, all_msgs, ... = await self._run_agent_loop(...)

save_skip = 1 + len(history) + (1 if user_persisted_early else 0)
self._save_turn(session, all_msgs, save_skip)
self._clear_runtime_checkpoint(session)
self.sessions.save(session)
```

runner 返回的 `all_msgs` 是**完整消息列表**（系统提示词 + 历史 + 本轮新消息），`_save_turn` 通过 `save_skip` 切片只取本轮新生成的部分。

### save_skip 的含义

位置：[loop.py:1163-1167](agent/loop.py#L1163-L1167)

```python
def _save_turn(self, session, messages, skip):
    for m in messages[skip:]:    # 只处理 skip 之后的消息
        # ... 截断大工具结果、清理 runtime context ...
        session.messages.append(entry)
```

`save_skip` 的构成：

```
save_skip = 1                # 系统提示词（messages[0] 永远是 system prompt）
          + len(history)      # 从 session 回放的历史消息数
          + (1 if user_persisted_early else 0)  # 如果用户消息已提前写入，多跳一条
```

```
all_msgs = [system] + [hist1, hist2, ...] + [user?, assistant, tool, ...]
           ←────── skip 这么多 ──────────→  ←── 真正持久化的 ──→
```

用一个整数偏移量切片，比逐条比对查重高效得多。

### \_save_turn 中的清理逻辑

除了截断大工具结果（超出 `max_tool_result_chars` 的截断），`_save_turn` 还做了几件事：

- **跳过空的 assistant 消息**：`role == "assistant"` 且无 content 且无 tool_calls → 丢弃（毒化后续上下文）
- **剥离 runtime context**：user 消息中以 `_RUNTIME_CONTEXT_TAG` 开头的块（含 session summary）不写入持久化
- **图片占位符替换**：base64 图片替换为 `[Image: path]` 文本
- **追加时间戳**：每条消息补充 `timestamp` 字段

### 完整时序图

```
_process_message()
  │
  ├─ 1. session.add_message("user", ...)     ← 用户消息提前写盘
  │    sessions.save(session)
  │
  ├─ 2. _run_agent_loop(initial_messages)
  │    │
  │    ├─ runner iteration 1:
  │    │    ├─ LLM 返回 tool_calls
  │    │    ├─ _emit_checkpoint("awaiting_tools")   ← checkpoint 写 metadata
  │    │    ├─ 执行工具
  │    │    └─ _emit_checkpoint("tools_completed")  ← checkpoint 写 metadata
  │    │
  │    ├─ runner iteration N:
  │    │    ├─ LLM 返回 final text
  │    │    └─ _emit_checkpoint("final_response")   ← checkpoint 写 metadata
  │    │
  │    └─ 返回 all_msgs（完整消息列表）
  │
  ├─ 3. _save_turn(session, all_msgs, save_skip)   ← 批量追加到 session.messages
  │    _clear_runtime_checkpoint(session)
  │    sessions.save(session)                       ← 最终落盘
  │
  └─ 4. _schedule_background(
          consolidator.maybe_consolidate_by_tokens()  ← 后台异步触发记忆压缩
      )
```

### 特殊情况：/stop 中断恢复

当用户执行 `/stop` 时，task 被 `asyncio.CancelledError` 取消（[loop.py:786-811](agent/loop.py#L786-L811)）。catch 块中：

1. 调用 `_restore_runtime_checkpoint(session)` —— 把 checkpoint 中的部分消息物化到 messages
2. 调用 `_clear_pending_user_turn(session)`
3. `sessions.save(session)`

这样下次对话时，中断前已完成的消息（assistant + 已执行的 tool results）不会丢失，未完成的 tool_calls 会被补上 error 占位，LLM 看到这些上下文后可以合理继续。而对于正常结束的循环，`_clear_runtime_checkpoint` 会清掉 metadata 中的 checkpoint，不会残留。

## 6. AgentLoop 和 AgentRunner 的区别

### AgentLoop（产品层 / 编排层）

**位置**：[agent/loop.py](agent/loop.py)

是整个 agent 进程的**主循环**，负责"什么时候跑、为谁跑、跑完怎么收尾"：

| 职责 | 具体内容 |
|------|----------|
| 消息总线消费 | `run()` 死循环通过 `self.bus.consume_inbound()` 持续消费入站消息，分发给对应 session |
| 会话管理 | 通过 `SessionManager` 管理会话生命周期，per-session `asyncio.Lock` + `asyncio.Semaphore` 控制并发，pending queue 处理 mid-turn 注入 |
| 上下文构建 | 持有 `ContextBuilder`，组装 system prompt（identity + bootstrap files + memory + skills + history） |
| 工具注册 | 持有 `ToolRegistry`，注册所有内置工具（read_file、exec、grep、web_search 等） |
| 子代理管理 | 持有 `SubagentManager`，子 agent 的 `max_iterations` 通过 `_sync_subagent_runtime_limits()` 与主 loop 保持同步 |

### AgentRunner（引擎层 / 纯执行循环）

**位置**：[agent/runner.py](agent/runner.py)

是一个**无状态的单次 agent 运行引擎**，负责"跑起来之后每一步怎么跑"：

| 职责 | 具体内容 |
|------|----------|
| LLM + Tool 循环 | `for iteration in range(max_iterations)` → 调 LLM → 执行 tool calls → 追加结果 → 继续 |
| 上下文治理 | micro-compact（压缩旧的 tool 结果）、history snipping（超出 context window 时裁剪）、orphan tool result 清理、tool result 长度预算管理 |
| Tool 执行 | 支持并发/串行两种模式，处理 `AskUserInterrupt` |
| 容错重试 | 空响应重试（最多 2 次）、length recovery（最长回复被截断时重试 3 次） |
| Mid-turn 注入 | 通过 `injection_callback` 在工具执行间隙注入新的 user message（上限 3 条/轮，5 轮注入循环） |
| Checkpoint 发射 | 在 tool 执行前后发射 checkpoint 用于崩溃恢复（`awaiting_tools` → `tools_completed` → `final_response`） |
| Finalization 重试 | 循环结束仍无有效输出时，额外发送一条 "Please provide your final answer" 做最后尝试 |

**关键：Runner 不感知 session/key/channel**。入参是 `AgentRunSpec`（纯数据），出参是 `AgentRunResult`（纯数据），它只处理 messages 列表，不知道消息从哪来、要发到哪去。

### 入参/出参对比

```python

    max_tool_result_chars: int        # 工具结果截断上限

    injection_callback: ...           # mid-turn 注入回调

# AgentRunner 的出参 —— 纯结果
@dataclass

    messages: list[dict]              # 完整消息列表

```

### 调用关系

```
AgentLoop.run()
  └─ while self._running:
       └─ msg = await bus.consume_inbound()
            └─ asyncio.create_task(_dispatch(msg))
                 └─ _process_message(msg)
                      ├─ 1. 用户消息提前写盘 (session.add_message + save)
```

### 一句话总结

**AgentLoop = Manager + Scheduler**，管"什么时候跑、为谁跑、跑完怎么收尾"；**AgentRunner = Stateless Engine**，管"跑起来之后每一步怎么跑"。Loop 是 Runner 的唯一调用方，Runner 不感知任何产品层概念。

### 路由器：三层分发 + 拦截器

[command/router.py:27-98](command/router.py#L27-L98) 中 `CommandRouter` 是一个纯字典驱动的命令分发器：

```
CommandRouter
├── _priority: dict[str, Handler]       ← 精确匹配，dispatch lock 之外执行
├── _exact: dict[str, Handler]          ← 精确匹配，dispatch lock 之内执行
├── _prefix: list[(str, Handler)]       ← 最长前缀优先（如 "/history " 匹配 "/history 5"）
└── _interceptors: list[Handler]        ← 兜底谓词拦截器（当前未使用）
```

注册方式对应三个方法：

```python
router.priority("/stop", cmd_stop)        # priority 层
router.exact("/new", cmd_new)             # exact 层
router.prefix("/history ", cmd_history)   # prefix 层（注意尾部空格）
```

`/status` 同时注册了 priority 和 exact 两层——保证无论走快速通道还是正常 dispatch 都能命中。`/history`、`/dream-log`、`/dream-restore` 同时注册 exact（无参数）和 prefix（带参数），覆盖两种调用场景。

### Context 结构

[command/router.py:16-24](command/router.py#L16-L24)：

```python

    msg: InboundMessage          # 原始入站消息
```

`loop` 引用是命令能操作 agent 内部状态的关键——`cmd_stop` 通过 `loop._cancel_active_tasks()` 取消任务，`cmd_new` 通过 `loop.sessions` 清空并重建 session。

### 执行流程：两条路径

命令分发在 agent loop 中有两条独立路径，分别服务于不同的时效性要求。

#### 路径一：run() 主循环中提前拦截（高优先级）

位置：[agent/loop.py:677-696](agent/loop.py#L677-L696)

```
run() 主循环消费到消息
  │
  ├─ is_priority(raw)? ──是──→ _dispatch_command_inline()
  │   │                          └─ dispatch_priority() → handler(ctx) → publish 结果
  │   │                          └─ continue（跳过 LLM，不进入 _process_message）
  │   │
  │   │  # 设计意图：/stop、/restart 必须在 dispatch lock 外执行，
  │   │  # 即使当前 session 正被 LLM 推理占用也能立即响应
  │   │
  │   └─（否，继续）
  │
  ├─ 会话已有活跃队列（正在处理消息）?
  │   ├─ is_dispatchable_command(raw)? ──是──→ _dispatch_command_inline()
  │   │   └─ dispatch() → exact/prefix 匹配执行 → publish → continue
  │   │
  │   └─ 否 → 作为 mid-turn 注入排队（稍后由 runner 的 injection_callback 消费）
  │
  └─ 普通消息 → 进入 _process_message()
```

**设计要点**：

- **priority 命令不需要 session**：`_dispatch_command_inline` 中 `ctx.session = None`，因为 `/stop`、`/restart` 的语义是进程级的，不依赖当前会话状态
- **mid-turn 注入不能是命令**：如果 session 已有活跃处理队列，非命令消息会排队等待注入，但命令消息（`is_dispatchable_command` 返回 True）会**直接执行**而非排队——防止命令被阻塞在队列中迟迟得不到处理

#### 路径二：_process_message() 内部最后一道拦截

位置：[agent/loop.py:976-980](agent/loop.py#L976-L980)

```
_process_message(msg)
  │
  ├─ 1. 用户消息提前写盘
  ├─ 2. 拼装上下文（identity + memory + skills + history）
  ├─ 3. 检查是否为命令 ←── 最后一道拦截
  │      ctx = CommandContext(msg=msg, session=session, key=key, raw=raw, loop=self)
  │      if result := await self.commands.dispatch(ctx):
  │          return result              # 命中命令，不进入 LLM
  │
  ├─ 4. maybe_consolidate_by_tokens()   # 压缩检查
  └─ 5. _run_agent_loop()              # 进入 LLM 推理
```

dispatch 的匹配顺序：`exact → prefix → interceptors`，任一命中即返回，不再继续。

### 各命令的实现要点

**`/stop`** [command/builtin.py:16-25](command/builtin.py#L16-L25)
- 调用 `loop._cancel_active_tasks(session_key)` 取消该 session 下所有 asyncio.Task
- 取消操作是被协作式的——task 内的 `asyncio.CancelledError` catch 块负责收尾（物化 checkpoint、保存 session）
- 返回被取消的任务数量

**`/restart`** [command/builtin.py:28-45](command/builtin.py#L28-L45)
- 先将重启通知写入环境变量（`set_restart_notice_to_env`），进程重启后 CLI 层读取该变量，把 "Restarting..." 消息重放到对应 channel

- `os.execv` 会完全替换当前进程，Python 解释器重新初始化，所有内存状态丢失，config 和 cron 从磁盘重新加载

**`/status`** [command/builtin.py:48-95](command/builtin.py#L48-L95)

- 搜索用量通过 `fetch_search_usage` 异步获取，但用 try/except 包裹，失败不影响整体响应
- `render_as: "text"` 标记让 channel 层跳过 markdown 渲染

**`/new`** [command/builtin.py:98-113](command/builtin.py#L98-L113)

## 8. Runner 上下文治理：_snip_history() vs Consolidator

两个机制的目标都是防止超出 LLM 上下文窗口，但手段完全不同：**一个是运行时临时截断，一个是持久化 LLM 摘要压缩**。

### 一句话总结

- **`_snip_history()`**：每次调 LLM 前的**机械兜底**——纯 token 计数，从前面裁掉旧消息，只影响发给 LLM 的副本，不写盘
- **Consolidator**：对话结束后的**智能压缩**——调 LLM 把旧消息总结成摘要，永久替换原始消息，摘要注入后续对话

### 逐项对比

| 维度 | `_snip_history()` | Consolidator |
|------|-------------------|--------------|
| **位置** | [runner.py:1025-1092](agent/runner.py#L1025-L1092) | [memory.py:443-692](agent/memory.py#L443-L692) |
| **触发时机** | 每次迭代调 LLM 前（每轮都跑） | `_process_message` 完成后，`_schedule_background` 异步触发 |
| **作用对象** | `messages_for_model`（`messages` 的派生副本） | `session.messages`（持久化的原始消息列表） |
| **操作方式** | 纯 token 计数，FIFO 从前面裁掉旧消息 | 调用独立 LLM 将旧消息**总结**为一段文本 |
| **持久化** | **不写盘**——原始 `messages` 列表不变，loop 层最终照常全量 `_save_turn()` | **写盘**——摘要写入 `history.jsonl`，`session.last_consolidated` 游标前移 |
| **智能程度** | 零，纯粹看 token 数 | LLM 理解对话语义，产出有意义的摘要 |
| **数据可逆** | 临时的，下次调用时原始 `messages` 仍在 | **不可逆**，原始消息被永久删除，只剩摘要 |
| **额外 LLM 调用** | 无 | 有（每次 archive 都要调一次 LLM） |
| **作用范围** | 单次 runner 执行内部 | 跨轮次、session 级别 |

### _snip_history()：机械兜底

[runner.py:1025-1092](agent/runner.py#L1025-L1092)，每轮迭代在调 LLM 之前执行。

#### 预算计算

```python
budget = context_window_tokens - max_output_tokens - 1024  # 1024 为安全缓冲区
# 例：128K - 8K - 1K = 119K token 可用于历史消息
```

#### 裁剪逻辑

```
估算 messages_for_model 的 token 数
  ├─ ≤ budget → 原样返回（无需裁剪）
  └─ > budget →
        ├─ 1. 保留所有 system 消息（提示词一条不丢）
        ├─ 2. 分离非 system 消息
        ├─ 3. 从尾部倒着扫描，累积到接近 budget 即停止
        ├─ 4. 确保第一条非 system 消息是 user（API 兼容性：部分模型拒绝 system→assistant 的顺序）
        └─ 5. find_legal_message_start() 调整起点，避免以 tool 消息开头
```

#### 在治理流水线中的位置

```
原始 messages（可能远超窗口）
  │
  ├─ drop_orphan_tool_results      → 清理孤儿 tool 结果
  ├─ backfill_missing_tool_results → 补全缺失的 tool 结果
  ├─ microcompact                  → 旧 tool 结果压缩为一行摘要
  ├─ apply_tool_result_budget     → 超出 max_tool_result_chars 的截断
  ├─ snip_history                  → ★ 超出窗口时从前面裁剪
  ├─ drop_orphan_tool_results      → 裁剪后再次清理
  └─ backfill_missing_tool_results → 裁剪后再次补全
       │
       ▼
  messages_for_model（发给 LLM 的安全版本）
```

#### 关键代码注释

```python
# 原始 messages 保持不变。Context governance 可能修复或压缩历史消息，
# 但这些合成编辑不能移动调用方用于保存新轮次的追加边界。
messages_for_model = self._drop_orphan_tool_results(messages)
messages_for_model = self._microcompact(messages_for_model)
# ...
messages_for_model = self._snip_history(spec, messages_for_model)
```

这保证了 **`messages` 列表毫发无损**，loop 层最终调用 `_save_turn()` 时能正确通过 `save_skip` 偏移量切片出本轮新增的消息。

### Consolidator：持久化智能压缩

[memory.py:443-692](agent/memory.py#L443-L692)，在 `_process_message` 完成后触发。

#### 触发条件

```python
async def maybe_consolidate_by_tokens(session):
    budget = context_window_tokens - max_completion_tokens - 1024
    estimated, _ = self.estimate_session_prompt_tokens(session)
    if estimated < budget:
        return  # 没超，不需要压缩
    # 超出预算 → 开始压缩
```

#### 压缩流程

```
session.messages（持久化消息列表，已累积数百条）
  │
  ├─ 1. 估算当前 prompt token 量
  │
  ├─ 2. pick_consolidation_boundary(tokens_to_remove)
  │      └─ 从 session.last_consolidated 开始扫描
  │      └─ 只在 user 消息前切分（保证不把一轮对话切碎）
  │      └─ 累积到 tokens_to_remove 时返回边界索引
  │
  ├─ 3. archive(chunk) — 取 last_consolidated 到边界之间的消息
  │      ├─ 格式化消息为文本（timestamp + role + content）
  │      ├─ 调用独立 LLM（consolidator_archive.md 模板）
  │      │     "Summarize the following conversation messages..."
  │      ├─ LLM 返回摘要文本
  │      └─ 摘要写入 history.jsonl（持久化存储）
  │
  ├─ 4. session.last_consolidated = end_idx  ← 游标前移
  ├─ 5. sessions.save(session)               ← 落盘
  │
  └─ 6. 最多 5 轮循环（_MAX_CONSOLIDATION_ROUNDS），
         直到 token 预算降到 target 以下
```

#### 摘要的后续利用

压缩完成后，最后一个摘要会存入 `session.metadata["_last_summary"]`。下一次 `_process_message` 调用 `prepare_session()` 时，这个摘要会作为 **runtime context** 注入到系统提示词中，让 LLM 知道"之前聊过什么"，即便原始消息已被清除。

#### 降级策略

如果 LLM 摘要调用失败，`archive()` 会降级到 `raw_archive()`——直接把原始消息格式化后写入 `history.jsonl`，前面加 `[RAW]` 标记。虽然质量不如 LLM 摘要，但至少不会静默丢失对话记录。

### 两层防线的协作关系

```
新消息到达 _process_message()
  │
  ├─ 第一道防线：Consolidator
  │   maybe_consolidate_by_tokens(session)
  │   ├─ 如果 session 消息过多 → LLM 摘要压缩 → last_consolidated 前移
  │   └─ 保证进入 runner 的 initial_messages 已在大致安全范围内
  │
  ├─ context_builder.build_messages()
  │   └─ session.get_history() 从 last_consolidated 开始取消息
  │      → 已被 consolidate 的消息不参与构建（被摘要替代）
  │
  └─ runner.run(initial_messages)
        │
        └─ 第二道防线：_snip_history()
            ├─ 每轮迭代前执行
            ├─ 即使 Consolidator 没压够（一轮对话产生了大量 tool 结果）
            ├─ 或在 Consolidator 来不及跑的快速连续对话场景
            └─ 仍然做最后的 token 计数兜底，保证 LLM 不超窗口
```

### 为什么需要两层？

1. **Consolidator 是批量操作，有滞后性**：它在对话结束后触发，如果用户在一轮里跑了 180 个 tool iteration，消息暴涨，Consolidator 来不及救场
2. **Consolidator 成本高**：每次 archive 要额外调一次 LLM，不能每轮都跑，只能在必要时批量处理
3. **`_snip_history` 是终极保险**：纯 token 计算，零额外成本，每轮必跑——即使前面所有治理都失效，它也能保证 LLM 不会收到超出窗口的消息列表

简言之：**Consolidator = 日常深度压缩（持久化 + 语义理解），`_snip_history` = 每轮兜底截断（临时 + 纯数学）**。

## 9. 主 Agent 何时调用 Subagent？——Prompt 层面的指导缺失

核心发现：**主 agent 关于"何时使用 subagent"的指导几乎为零**——只有 `spawn` 工具自身 4 句 description，所有 prompt 模板均未涉及 subagent 使用策略。

### 当前唯一的指导

[agent/tools/spawn.py:40-47](agent/tools/spawn.py#L40-L47)，`SpawnTool.description` 属性，这是 LLM 在 function calling 的 `tools` 数组中看到的工具描述：

```python
"Spawn a subagent to handle a task in the background. "
"Use this for complex or time-consuming tasks that can run independently. "
"The subagent will complete the task and report back when done. "
"For deliverables or existing projects, inspect the workspace first "
"and use a dedicated subdirectory when helpful."
```

翻译为决策逻辑就是四种情况：

| 条件 | 含义 |
|------|------|
| "complex or time-consuming" | 任务复杂或耗时 → 考虑用 |
| "can run independently" | 任务不依赖当前对话的中间状态 → 可以派发 |
| "report back when done" | 完成后子 agent 的回复会自动推送 → 主 agent 不用轮询 |
| "inspect workspace first" | 与已有项目交互时先看工作区 → 别盲派 |

### 各 Prompt 模板的覆盖情况

| 模板文件 | 是否涉及 subagent | 当前内容 |
|---------|------------------|---------|
| [SOUL.md](templates/SOUL.md) | **否** | "单步立即执行，多步先列计划"，但不区分"自己做"vs"派发" |
| [TOOLS.md](templates/TOOLS.md) | **否** | 只覆盖 exec、glob、grep、cron，**无 spawn 条目** |
| [AGENTS.md](templates/AGENTS.md) | **否** | 只讲 cron 和 heartbeat，未涉及 subagent 策略 |
| [context.py](agent/context.py) | **否** | 上下文构建时未注入任何 subagent 策略 |
| [subagent_system.md](templates/agent/subagent_system.md) | 仅子 agent 视角 | 告诉子 agent "你是被派发的，专注任务，结果回报"，不是给主 agent 的 |

### 缺失的指导维度

对比 Claude Code 的 subagent 系统提示词（对 `Task` 工具有详细的使用场景说明），nanobot 缺少以下关键指导：

#### 1. 什么情况**应该**用 subagent？

```
缺失：
- 搜索范围广、涉及 3 个以上文件的代码探索
- 需要独立上下文窗口的深度分析（避免污染主对话）
- 多个互不依赖的任务可以并行派发
- 长时间运行的批处理操作
```

#### 2. 什么情况**不应该**用 subagent？

```
缺失：
- 简单单步操作（读一个已知路径的文件 → 自己读）
- 依赖当前对话中刚生成的中间结果（子 agent 看不到）
- 需要用户实时确认的交互式操作
- 纯粹的信息查询（LLM 常识能直接回答）
```

#### 3. 并行派发策略

```
缺失：
- 多个独立任务应一次性发出多个 spawn 调用
- 互有依赖的任务必须串行（等前一个完成再派下一个）
- 同一 session 下活跃 subagent 数量限制
```

#### 4. 失败处理

```
缺失：
- 子 agent 超时或报错后，主 agent 应如何处理？
- 哪些错误值得重试？哪些应该换个方式自己来？
- 子 agent 返回空结果时的判断标准
```

#### 5. 工具能力差异

```
缺失（主 agent 不知道子 agent 能力边界）：
- 子 agent 有相同的 tool set（exec、read_file、grep 等）但独立运行
- 子 agent 看不到主 agent 的 conversation history
- 子 agent 的 max_iterations 受主 agent 同步控制
```

### LLM 的"常识"能补多少？

当前 nanobot 把判断权完全交给 LLM 的通用推理能力。实际效果取决于模型：

| 场景 | 强模型（Opus 4.7 / GPT-5） | 弱模型（Haiku / 小参数模型） |
|------|---------------------------|---------------------------|
| 明确说"帮我搜一下全项目的 XXX" | 可能自发 spawn Explore agent | 可能自己逐个 grep，效率低 |
| 用户给了 5 个独立任务 | 可能并行 spawn 5 个 | 可能逐个做，甚至不做完 |
| 简单问题如"1+1 等于几" | 直接回答，不用 subagent | 同样直接回答 |
| 边界模糊的任务 | 可能正确判断 | 可能误 spawn（浪费）或该 spawn 不 spawn（慢） |

### 改进方向

可以从三个层面增强：

**方案一：TOOLS.md 补条目（成本最低）**
```markdown
## spawn — Background Subagents

- Use for tasks that require searching 3+ files or spanning multiple directories
- Do NOT use for single-file reads, simple math, or pure knowledge questions
- Multiple independent tasks → spawn them in parallel
- Task depends on current conversation context → do it yourself
- Subagent failed? Read the error, adjust the task description, and retry once
```

**方案二：SOUL.md 加执行策略**
```markdown
## Delegation Rules
- If a task requires broad codebase exploration, spawn a subagent.
- If a task has clear scope and can finish in < 3 tool calls, do it yourself.
- When given N independent tasks, spawn N subagents in parallel.
```

**方案三：context.py 注入 runtime 策略（最灵活）**
在 `_build_messages` 中根据 session 状态动态注入 subagent 使用提示——例如当前已有 3 个活跃 subagent 时提醒节制，或检测到并行任务场景时显式建议批量 spawn。

# JSONL跟JSON的区别

JSONL内存友好，不需要像JSON那样一次把所有文件加载进内存。
JSONL更方便追加数据。
JSON容错性更好，即使最后一行数据破损，前面的数据依然可以正常解析。
