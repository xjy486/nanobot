# 工具系统架构梳理

## 概述

本文梳理 nanobot 项目中工具的加载、注册、执行机制，重点对比 **内置工具** 与 **MCP 工具** 的区别。

---

## 一、核心抽象：Tool 基类

所有工具（无论内置还是 MCP）都继承自统一的抽象基类 `Tool`，位于 [`agent/tools/base.py`](agent/tools/base.py:117)。

```python
class Tool(ABC):
    @property
    @abstractmethod
    def name(self) -> str: ...

    @property
    @abstractmethod
    def description(self) -> str: ...

    @property
    @abstractmethod
    def parameters(self) -> dict[str, Any]: ...

    @property
    def read_only(self) -> bool:
        return False  # 是否无副作用，影响并发安全

    @property
    def exclusive(self) -> bool:
        return False  # 是否独占执行

    @property
    def concurrency_safe(self) -> bool:
        return self.read_only and not self.exclusive

    @abstractmethod
    async def execute(self, **kwargs: Any) -> Any: ...
```

### 关键设计

| 属性               | 含义                                   | 影响                                 |
| ------------------ | -------------------------------------- | ------------------------------------ |
| `read_only`        | 工具是否无副作用（如只读文件）         | 决定是否可以与其他工具并发执行       |
| `exclusive`        | 工具是否需要独占执行                   | 即使是 read_only，独占工具也不能并发 |
| `concurrency_safe` | 综合判断 `read_only and not exclusive` | 用于批次分区，决定哪些工具可以并行   |

### 参数定义装饰器

内置工具使用 `@tool_parameters` 装饰器（[`base.py:246`](agent/tools/base.py:246)）直接附加 JSON Schema，避免手写 `property`：

```python
@tool_parameters({
    "type": "object",
    "properties": {"path": {"type": "string"}},
    "required": ["path"],
})
class ReadFileTool(Tool): ...
```

## 二、工具注册表

`ToolRegistry` 位于 [`agent/tools/registry.py`](agent/tools/registry.py:8)，是工具的核心管理容器：

```python
class ToolRegistry:
    def __init__(self):
        self._tools: dict[str, Tool] = {}
        self._cached_definitions: list[dict] | None = None

    def register(self, tool: Tool) -> None:
        self._tools[tool.name] = tool
        self._cached_definitions = None  # 清除缓存

    def get(self, name: str) -> Tool | None:
        return self._tools.get(name)

    async def execute(self, name: str, params: dict) -> str:
        """根据名字执行工具"""
        tool = self._tools.get(name)
        if not tool:
            return f"Error: Unknown tool '{name}'"
        try:
            return await tool.execute(**params)
        except Exception as e:
            return f"Error executing {name}: {e}"
```

### 工具定义排序（缓存友好）

[`get_definitions()`](agent/tools/registry.py:48) 方法对工具进行稳定排序：

1. **内置工具** 排在前面，按名称排序
2. **MCP 工具** 排在后面，按名称排序

这样 LLM 的 system prompt 中工具列表顺序稳定，有利于 prompt caching。

### 调用准备与执行

```python
def prepare_call(self, name, params) -> tuple[Tool | None, dict, str | None]:
    # 1. 查找工具
    # 2. cast_params：参数类型转换（字符串转 int/bool 等）
    # 3. validate_params：JSON Schema 校验

async def execute(self, name, params) -> Any:
    tool, params, error = self.prepare_call(name, params)
    if error: return error + hint
    result = await tool.execute(**params)
    return result
```

```python
def _register_default_tools(self) -> None:
    # 1. 注册 ask_user
    self.tools.register(AskUserTool())

    # 2. 注册文件系统工具
    self.tools.register(ReadFileTool(workspace=..., allowed_dir=...))
    self.tools.register(WriteFileTool(...))
    self.tools.register(EditFileTool(...))
    self.tools.register(ListDirTool(...))

    if self.exec_config.enable:
        self.tools.register(ExecTool(...))

    # 4. 条件注册 web 工具
    if self.web_config.enable:
        self.tools.register(WebSearchTool(...))
        self.tools.register(WebFetchTool(...))

    # 5. 注册 message、spawn、cron
    self.tools.register(MessageTool(...))
    self.tools.register(SpawnTool(...))
    if self.cron_service:
        self.tools.register(CronTool(...))
```

### 3.3 特点

- **编译时确定**：工具类在代码中显式定义，项目启动时即存在
- **直接实例化**：`registry.register(XXXTool(...))`
- **本地执行**：`execute()` 直接调用 Python 方法，无网络/进程间开销
- **无连接管理**：与程序同生命周期，无需额外维护连接状态

## 四、MCP 工具的加载与注册

### 4.1 MCP 简介

MCP（Model Context Protocol）是 Anthropic 推出的开放协议，允许 AI 助手连接外部工具服务器。nanobot 作为 MCP Client，可以连接到任意 MCP Server 并使用其暴露的工具。

### 4.2 配置方式

MCP 服务器在 `config.yaml` 中配置，类型定义于 [`config/schema.py`](config/schema.py:218)：

```python
class MCPServerConfig(Base):
    type: Literal["stdio", "sse", "streamableHttp"] | None = None
    command: str = ""          # stdio 模式：启动命令
    args: list[str] = []
    env: dict[str, str] = {}
    url: str = ""              # HTTP/SSE 模式：服务端点
    headers: dict[str, str] = {}
    tool_timeout: int = 30
    enabled_tools: list[str] = ["*"]  # 工具白名单，默认全部启用
```

### 4.3 连接与注册流程

MCP 工具通过 [`agent/tools/mcp.py:connect_mcp_servers()`](agent/tools/mcp.py:436) 动态发现并注册：

```
AgentLoop.run() / process_direct()
  └── _connect_mcp()           ← 懒加载，首次需要时才连接
        └── connect_mcp_servers(mcp_servers, registry)
              └── 对每个服务器并行执行 connect_single_server()
                    ├── 1. 建立传输连接
                    │     ├── stdio: 启动子进程，通过 stdin/stdout 通信
                    │     ├── sse: Server-Sent Events over HTTP
                    │     └── streamableHttp: 流式 HTTP
                    ├── 2. 创建 ClientSession
                    │     └── session.initialize()
                    ├── 3. 获取并注册工具
                    │     └── session.list_tools()
                    │         └── MCPToolWrapper(session, ...) → registry.register()
                    ├── 4. 获取并注册资源
                    │     └── session.list_resources()
                    │         └── MCPResourceWrapper(...) → registry.register()
                    └── 5. 获取并注册提示词
                          └── session.list_prompts()
                              └── MCPPromptWrapper(...) → registry.register()
```

### 4.4 MCP 工具包装器

MCP 工具通过 `MCPToolWrapper` 包装成 nanobot 的 `Tool`，见 [`agent/tools/mcp.py:144`](agent/tools/mcp.py:144)：

```python
class MCPToolWrapper(Tool):
    def __init__(self, session, server_name, tool_def, tool_timeout=30):
        self._session = session
        self._original_name = tool_def.name
        self._name = _sanitize_name(f"mcp_{server_name}_{tool_def.name}")
        self._description = tool_def.description or tool_def.name
        self._parameters = _normalize_schema_for_openai(tool_def.inputSchema)

    async def execute(self, **kwargs) -> str:
        result = await asyncio.wait_for(
            self._session.call_tool(self._original_name, arguments=kwargs),
            timeout=self._tool_timeout,
        )
        # 提取文本结果
        parts = [block.text for block in result.content if isinstance(block, TextContent)]
        return "\n".join(parts)
```

### 4.5 MCP 资源与提示词

除了工具（Tool），MCP 协议还定义了两种额外的能力类型，nanobot 都将它们包装为只读 `Tool`：

**MCPResourceWrapper**（资源）：

- 通过 `session.read_resource(uri)` 读取
- 名称格式：`mcp_{server}_resource_{name}`
- `read_only = True`

**MCPPromptWrapper**（提示词）：

- 通过 `session.get_prompt(name, arguments)` 获取
- 名称格式：`mcp_{server}_prompt_{name}`
- `read_only = True`

### 4.6 名称处理

MCP 工具名称会被规范化（[`_sanitize_name`](agent/tools/mcp.py:37)）：

- 添加前缀：`mcp_<server_name>_<original_name>`
- 替换非法字符：`[^a-zA-Z0-9_-]` → `_`
- 合并连续下划线

例如：server `filesystem` 中的 `read_file` 会变成 `mcp_filesystem_read_file`。

### 4.7 错误处理与重试

MCP 工具执行包含完善的错误处理（[`MCPToolWrapper.execute`](agent/tools/mcp.py:168)）：

| 错误类型         | 处理方式                             |
| ---------------- | ------------------------------------ |
| `TimeoutError`   | 返回超时提示                         |
| `CancelledError` | 判断是否为外部取消，决定是否重新抛出 |
| 瞬态连接错误     | 等待 1 秒后重试一次                  |
| 其他异常         | 记录日志并返回错误信息               |

瞬态错误包括：`ClosedResourceError`、`BrokenResourceError`、`EndOfStream`、`BrokenPipeError`、`ConnectionResetError` 等。

### 4.8 懒加载与缓存细节

MCP 连接是**真正的"懒加载"**——`AgentLoop.__init__()` 从不调 `_connect_mcp()`，只保存配置。首次连接发生在第一条消息到达时：

```
AgentLoop.run() / process_direct()
  └── _connect_mcp()                          loop.py:395
        ├─ if _mcp_connected or _mcp_connecting or not _mcp_servers: return
        ├─ _mcp_connecting = True             ← 防止并发重复连接
        ├─ await connect_mcp_servers(...)
        │     └── 对每个 server:
        │           ├─ session.initialize()
        │           ├─ session.list_tools()    ← 网络请求，只调一次
        │           ├─ session.list_resources()
        │           └─ session.list_prompts()
        ├─ _mcp_connected = True              ← 至少一个成功才标记
        └─ _mcp_connecting = False
```

**三级缓存/防护：**

| 层级 | 机制 | 作用 |
|---|---|---|
| `_mcp_connecting` | 布尔锁 | 防止多个 task 同时尝试连接 |
| `_mcp_connected` | 布尔标记 | 连接成功后不再重复连接；失败则下次消息自动重试 |
| `_cached_definitions` | 内存缓存 | `get_definitions()` 每次 LLM 请求都调用，但只是返回缓存列表（O(1)），不重查 MCP 服务器 |

`_cached_definitions` 只在 `registry.register()` / `unregister()` 时失效，没有 TTL 过期。内置工具永远排在 MCP 工具前面，同组内按名称字母排序，保证工具列表跨请求稳定，有利于 LLM 端 prompt caching。

## 五、工具的执行流程

### 5.1 整体流程

```
AgentRunner.run()
  └── _request_model() → LLM 返回 tool_calls
        └── _execute_tools(tool_calls)
              ├── _partition_tool_batches()  ← 按并发安全分区
              └── 对每个批次：
                    ├── 并发批次：asyncio.gather() 并行执行
                    └── 串行批次：逐个执行 _run_tool()
                          ├── prepare_call()  ← 参数查找、类型转换、校验
                          └── tool.execute()  ← 实际执行
```

### 5.2 并发控制

工具调用的并发通过 [`_partition_tool_batches()`](agent/runner.py:1094) 实现：

```python
def _partition_tool_batches(self, spec, tool_calls):
    if not spec.concurrent_tools:
        return [[tc] for tc in tool_calls]  # 全串行

    batches = []
    current = []
    for tc in tool_calls:
        tool = spec.tools.get(tc.name)
        can_batch = tool and tool.concurrency_safe  # read_only and not exclusive
        if can_batch:
            current.append(tc)  # 加入并发批次
        else:
            if current:
                batches.append(current)
                current = []
            batches.append([tc])  # 独占工具单独执行
    ...
```

**例子**：假设 LLM 在一次响应中请求 4 个工具：

- `read_file` (read_only=True) → 可并发
- `glob` (read_only=True) → 可并发
- `exec` (read_only=False) → 独占
- `write_file` (read_only=False) → 独占

分区结果：`[[read_file, glob], [exec], [write_file]]`

即：`read_file` 和 `glob` 并行执行，然后串行执行 `exec`，最后串行执行 `write_file`。

### 5.3 单次工具执行

[`_run_tool()`](agent/runner.py:731) 的执行逻辑：

1. **重复查找检查**：防止 LLM 反复进行相同的外部查询
2. **prepare_call**：通过 `registry.prepare_call()` 查找工具、参数类型转换、JSON Schema 校验
3. **执行**：
   - 优先直接调用 `tool.execute(**params)`（如果 `prepare_call` 返回了 tool 实例）
   - 否则回退到 `registry.execute(name, params)`
4. **结果处理**：
   - 正常结果：返回结果内容
   - `Error` 开头：返回错误 + hint
   - workspace 违规：记录 warning 并中止当前轮次

### 5.4 多工具调用结果组装

当 LLM 在一次响应中同时调用 N 个工具时，结果以 **1 条 assistant + N 条 tool** 的形式拼入 messages（[`runner.py:290-336`](agent/runner.py#L290-L336)）：

```python
# 1 条 assistant 消息，包含 N 个 tool_call
assistant_message = build_assistant_message(
    content=response.content,
    tool_calls=[tc_A, tc_B, tc_C],      # 全部 N 个 tool_call
)
messages.append(assistant_message)       # 只追加 1 条

# N 条 tool 消息，逐个追加
for tool_call, result in zip(tool_calls, results):
    messages.append({
        "role": "tool",
        "tool_call_id": tool_call.id,    # 通过 id 与 assistant 的 tool_call 配对
        "name": tool_call.name,
        "content": result,
    })
```

最终 messages 形状：

```
[...
  {"role": "assistant", "content": "让我同时查几样东西", "tool_calls": [
      {"id": "call_1", "function": {"name": "read_file", ...}},
      {"id": "call_2", "function": {"name": "grep", ...}},
      {"id": "call_3", "function": {"name": "web_search", ...}},
  ]},
  {"role": "tool", "tool_call_id": "call_1", "name": "read_file",  "content": "def main():..."},
  {"role": "tool", "tool_call_id": "call_2", "name": "grep",       "content": "3 matches..."},
  {"role": "tool", "tool_call_id": "call_3", "name": "web_search", "content": "Results:..."},
]
```

**关键点**：LLM 通过 `tool_call_id` 把每条 tool 消息对应回 assistant 的 tool_call，不依赖消息顺序。`_drop_orphan_tool_results` 也正是利用这个 id 配对来清理孤立 tool 消息。

### 5.5 Microcompact：旧工具结果压缩

`_microcompact()`（[`runner.py:978`](agent/runner.py#L978-L1018)）在每个 ReAct 迭代开始时被调用，是 context governance 6 步加工链的第 3 步。

**问题场景**：排查 bug 时 LLM 可能连续调十几次 `read_file`/`grep`，到第 12 轮时上下文里塞了上千行早已不再需要的旧 tool 输出。

**策略**：保留最新的查询结果原文，更早的替换为占位摘要。

```
compactable tool results 总数 = 15
_MICROCOMPACT_KEEP_RECENT = 10      ← 保留最新 10 条完整内容
stale = 前 5 条                      ← 替换为占位符

替换前: {"role": "tool", "name": "read_file", "content": "def main():\n    ...800行..."}
替换后: {"role": "tool", "name": "read_file", "content": "[read_file result omitted from context]"}
```

**压缩范围**只针对"查询型"工具：

```python
_COMPACTABLE_TOOLS = {"read_file", "exec", "grep", "glob",
                      "web_search", "web_fetch", "list_dir"}
```

这些工具的共同特点：结果可能很大，但对后续推理的价值随时间衰减。`write_file`、`edit_file`、`ask_user` 等操作型工具不在压缩列表中——写入成功/失败、用户回答有持久意义。

**两个门槛**：

| 条件 | 含义 |
|---|---|
| compactable 结果 ≤ 10 条 | 信息量不大，不触发压缩 |
| 单条结果 < 500 字符 | 替换省不了几个 token，保留原文 |

## 六、MCP 工具的并发安全性

MCP 的三种 Wrapper 对 `read_only` 的处理不同（[`base.py:154-162`](agent/tools/base.py#L154-L162)）：

| Wrapper | `read_only` | `exclusive` | `concurrency_safe` | 能否并发 |
|---|---|---|---|---|
| `MCPToolWrapper`（普通工具） | `False`（默认） | `False`（默认） | `False` | ❌ 不能 |
| `MCPResourceWrapper`（资源） | `True`（显式，[mcp.py:257](agent/tools/mcp.py#L257)） | `False`（默认） | `True` | ✅ 能 |
| `MCPPromptWrapper`（提示词） | `True`（显式，[mcp.py:361](agent/tools/mcp.py#L361)） | `False`（默认） | `True` | ✅ 能 |

普通 MCP 工具默认 `read_only=False` 的原因：框架无法判定一个未知 MCP 服务器的工具是否有副作用（比如它可能是个 `delete_database`），保守策略一律串行。只有 MCP 的 Resource 和 Prompt（规范定义就是只读的）才标记为 `read_only=True`。

## 七、ask_user 工具：人机协同中断机制

`ask_user` 与其他工具截然不同——它不执行任何操作，而是**中断 ReAct 循环，等待人类输入**（[agent/tools/ask.py](agent/tools/ask.py)）。

### 机制流程

```
LLM 调用 ask_user(question="用 React 还是 Vue？", options=["React", "Vue"])
    │
    ▼
AskUserTool.execute()  →  raise AskUserInterrupt(question, options)    ask.py:53
    │                         ↑ 不返回结果，而是抛异常！
    ▼
runner._execute_tools() →  捕获 AskUserInterrupt
    │                       stop_reason = "ask_user"                   runner.py:338
    │                       break → 退出 ReAct 循环
    ▼
loop._process_message() →  ask_user_outbound() 生成带按钮的回复        loop.py:1108
    │                       用户看到问题 + 选项按钮
    │
    │  ... 用户点击 "React"（或输入文本）...
    │
    ▼
loop._process_message() →  pending_ask_user_id(history) 检测到未回答的 ask_user
    │                       → 走 ask_user_tool_result_messages()        loop.py:1002
    │                       用户回复被包装为 tool_result，不是 user 消息
    ▼
runner.run() →  LLM 看到 {"role": "tool", "name": "ask_user", "content": "React"}
                就像 ask_user 是一个普通工具，它的"结果"就是用户的回答
                继续执行原任务
```

### 关键设计

- **`exclusive = True`**（[ask.py:49](agent/tools/ask.py#L49)）：调用 ask_user 时不能同时调其他工具——不可能一边问用户一边写文件
- **抛异常而非返回值**：`execute()` 不返回字符串，而是 `raise AskUserInterrupt`。Runner 捕获后立即退出循环，不做任何其他处理
- **下一轮作为 tool_result 注入**：`pending_ask_user_id()` 扫描 history，找到未完成的 ask_user tool_call，将用户回复构造成 `{"role": "tool", "tool_call_id": ..., "name": "ask_user", "content": "用户回答"}`
- **与 message 工具的区别**：`message` 是非阻塞通知（发了继续跑），`ask_user` 是阻塞等待（必须等人回答）

## 八、内置工具 vs MCP 工具对比

| 特性         | 内置工具                         | MCP 工具                                |
| ------------ | -------------------------------- | --------------------------------------- |
| **定义位置** | `agent/tools/*.py`               | 外部 MCP 服务器                         |
| **继承关系** | 直接继承 `Tool`                  | 通过 `MCPToolWrapper` 包装              |
| **注册方式** | 代码显式 `registry.register()`   | 自动从 MCP 服务器发现并注册             |
| **注册时机** | `AgentLoop.__init__()`           | `run()` 或 `process_direct()` 时懒加载  |
| **名称格式** | 无前缀（如 `read_file`）         | `mcp_<server>_<tool>`                   |
| **参数定义** | Python `@tool_parameters`        | 从 MCP 的 JSON Schema 转换              |
| **执行方式** | 本地 Python 方法                 | MCP 协议调用外部进程/服务               |
| **传输协议** | 无（本地）                       | stdio / SSE / streamableHttp            |
| **连接管理** | 无需管理                         | 维护 `AsyncExitStack` + `ClientSession` |
| **错误处理** | 本地异常捕获                     | 含网络超时、瞬态错误重试                |
| **能力类型** | 仅工具                           | 工具 + 资源 + 提示词                    |
| **并发安全** | `read_only`/`exclusive` 属性控制 | `MCPToolWrapper` 默认不并发；`MCPResourceWrapper`/`MCPPromptWrapper` 只读可并发 |
| **配置方式** | 代码硬编码或配置开关             | `config.yaml` 中 `mcp_servers` 配置     |
| **生命周期** | 与程序同生命周期                 | 独立管理，需要显式关闭                  |
| **性能**     | 快（本地调用）                   | 有网络/进程间开销                       |

## 九、关键设计亮点

1. **统一抽象**：无论内置还是 MCP，最终都是 `Tool` 对象，对上层（LLM、Runner）完全透明
2. **懒加载 MCP**：首次 `run()` 时才连接 MCP 服务器，避免启动阻塞；连接失败会在下次消息时重试
3. **并发安全控制**：通过 `read_only` 和 `exclusive` 精细控制，只读工具自动并行，副作用工具串行
4. **稳定排序缓存**：`get_definitions()` 内置工具在前、MCP 工具在后，顺序稳定利于 prompt caching
5. **完善的错误处理**：MCP 工具含自动重试、超时处理、连接断开恢复；内置工具有 workspace 安全检查
6. **白名单过滤**：MCP 支持 `enabled_tools` 配置，可精确控制暴露哪些工具给 LLM
7. **Schema 规范化**：MCP 的 JSON Schema（含 `oneOf`/`anyOf`/nullable 等）被转换为 OpenAI 兼容格式
8. **Microcompact 增量压缩**：查询型工具的旧结果自动替换为占位摘要，保留最新 10 条完整内容，节省上下文 token
9. **人机协同中断**：`ask_user` 通过抛异常中断 ReAct 循环，用户回复作为 tool_result 注入继续执行，实现阻塞式人机交互
