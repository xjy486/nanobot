## 一、上下文窗口管理

### 要解决什么问题

LLM 有上下文窗口上限（128K tokens 看着大，但 system prompt、工具定义、技能内容已占掉不少），需要确保发给模型的 prompt 不超过窗口，同时尽量让模型掌握足够多的上下文。

### 怎么实现的

入口在 [agent/context.py](agent/context.py#L31)，`ContextBuilder.build_system_prompt()` 分层组装：

1. **身份模板** — 运行时信息（操作系统、Python 版本、工作区路径）、平台策略（Windows/macOS 差异）、不同渠道策略（如 Telegram、QQ、Discord 等的格式）
2. **引导文件** — SOUL.md、USER.md、AGENTS.md、TOOLS.md，全部注入
3. **长期记忆** — MEMORY.md 全量注入（非模板时）
4. **always 技能** — 如 memory 技能（历史检索指南），全量注入
5. **技能摘要** — 其他可用技能只注入摘要行（渐进加载，按需读取完整内容）
6. **Recent History** — history.jsonl 中 Dream 尚未处理的最新条目，最多 50 条 / 32K 字符

每条用户消息前注入运行时元数据（当前时间、频道、chat_id）。

#### 非 System Prompt 部分

System Prompt 只是上下文的一部分，完整的 LLM 输入还包含以下组件（全都消耗 token 预算）：

**1. Messages 数组** — [agent/context.py](agent/context.py#L132) `build_messages()` 组装：

```
[0] system      ← build_system_prompt() 产物
[1..N] history  ← 来自 chat_id.jsonl，Session.get_history() 加载
[N+1] user      ← 运行时元数据 + 用户消息（含图片 base64）合并为一条
```

会话历史加载链路：

```
SessionManager.get_or_create("channel:chat_id")
  → _load(key) → 读 sessions/<safe_key>.jsonl → 反序列化为 Session
  → Session.get_history(max_messages=120, max_tokens=<token 预算>)
    → 第一阶段：从 last_consolidated 切（已归档的不重复送）
    → 第二阶段：按消息数量截断，保留尾部最多 120 条
    → 第三阶段（max_tokens > 0 时）：按 token 预算从尾部二次过滤
    → 向前对齐 user 边界（不割裂 tool_call 链）
```

`max_tokens` 预算由 [agent/loop.py](agent/loop.py#L502) `_replay_token_budget()` 计算：`context_window - max_output - 1024`。

**2. 工具定义**（独立 API 参数，不计入 system prompt）：

[agent/tools/registry.py](agent/tools/registry.py#L48) `ToolRegistry.get_definitions()`：

- 遍历所有已注册 Tool → `tool.to_schema()` 生成 JSON Schema
- 内置工具按名排序在前，MCP 工具按名排序在后（顺序稳定，命中 Prompt Cache）
- 作为 API 调用 `tools` 参数传入，由 [agent/runner.py](agent/runner.py#L606-L609) `_build_request_kwargs()` 组装

**3. 运行时动态追加**（runner 循环中逐轮累加）：

```
assistant（含 tool_calls）→ tool 结果 × N → assistant（含 tool_calls）→ ...
→ assistant（纯文本，最终回复）
```

#### Token 预算覆盖范围

[agent/memory.py](agent/memory.py#L512) `Consolidator.estimate_session_prompt_tokens()` 用探针消息完整模拟 `build_messages()`，**以上所有组件全部纳入估算**：

```
estimate_session_prompt_tokens()
  → build_messages(history, "[token-probe]", ...)   ← 生成完整 messages 数组
  → estimate_prompt_tokens_chain(messages, tools)    ← 工具定义也计入
```

估算覆盖：system prompt + 会话历史 + 运行时元数据 + 当前消息 + 工具定义 JSON + framing overhead。**所有组件共享同一个 token 预算，超限都会触发压缩。**

### Token 估算

[utils/helpers.py](utils/helpers.py#L410) 的 `estimate_prompt_tokens_chain()`：

- 优先走 provider 自带 API（如 Anthropic `count_tokens`，最准）
- fallback 走 tiktoken（cl100k_base 编码），公式：文本内容 + 工具定义 JSON + reasoning_content + 每条消息 × 4（framing overhead）

### 面试怎么讲

> "我把上下文组装抽象成 ContextBuilder，按优先级分层注入：引导文件全量注入、技能摘要部分注入、近期历史按需注入。System Prompt 之外，会话历史从 chat_id.jsonl 按 token 预算截断加载，工具定义以 JSON Schema 独立传入，运行时元数据逐条注入。所有组件统一纳入 token 估算，超预算自动触发压缩，保证输入不超模型窗口上限。"

## 二、双层记忆设计

### 要解决什么问题

- LLM 跨会话无记忆，每次都是全新对话
- 全量注入历史太贵（token 消耗大），完全不记等于每次重新教
- 需要一个"记住关键信息但不过度消耗 token"的方案

### 第一层：知识文件（长期记忆）

三个文件，职责分离，全量注入 System Prompt：

| 文件          | 职责                           | 管理策略             |
| ------------- | ------------------------------ | -------------------- |
| **SOUL.md**   | 机器人性格、沟通风格、行为边界 | 永久，仅在错误时校正 |
| **USER.md**   | 用户身份、偏好、技术栈         | 永久，仅在变化时更新 |
| **MEMORY.md** | 项目上下文、重要决策、关键事件 | 动态管理，Dream 处理 |

**注入方式**：`ContextBuilder.build_system_prompt()` 始终加载并注入 System Prompt。

**为什么用 Markdown 而不是数据库**：零依赖、人类可读可编辑、Agent 天然理解 Markdown、Git 可追溯。

### 第二层：history.jsonl（事件时间线）

存储位置：`<workspace>/memory/history.jsonl`

**格式**：每行一个 JSON 对象，三个字段：

```json
{
  "cursor": 1,
  "timestamp": "2025-05-03 14:30",
  "content": "用户讨论了上下文窗口管理方案"
}
```

**核心特性**：

- **仅追加模式**（Append-Only）：自增 cursor，`MemoryStore.append_history()` 追加写入
- **不整体注入上下文**：只有 Dream 未处理的最新 50 条作为 "Recent History" 注入
- **grep 按需检索**：Agent 通过 grep 工具搜索历史，memory 技能（always 加载）提供检索指南
- **内容安全处理**：strip_think 移除模型推理标签，每条最大 64K 字符硬上限
- **大小控制**：compact_history() 超 1000 条时丢弃最旧条目

### Dream 两阶段记忆处理器

由 cron 定时触发（默认每小时），消费 history.jsonl 中未处理的条目，更新知识文件。

**游标机制**：history.jsonl 每条有自增 cursor，Dream 通过 `.dream_cursor` 文件记录已处理到哪里，只处理新增条目。

**Phase 1 — 分析阶段（纯 LLM，无工具）**：

- 输入：待处理 history.jsonl 条目 + 当前 SOUL.md / USER.md / MEMORY.md 内容
- MEMORY.md 行级年龄标注：通过 `git blame` 计算每一行的最后修改天数，行尾追加 `← Nd`，帮助 LLM 判断内容是否过时（仅 MEMORY.md 做标注，SOUL.md / USER.md 不做）
- 输出：`[FILE]`(新增) / `[FILE-REMOVE]`(删除) / `[SKILL]`(建议创建技能) / `[SKIP]`(无需操作)

**Phase 2 — 编辑阶段（AgentRunner + 工具）**：

- 配备 read_file / edit_file / write_file 工具
- 精确增量编辑，不整体重写文件
- 同一文件多变更合并为一次 edit_file
- 最多 10 轮迭代，fail_on_tool_error=False 容错

**Git 版本控制**：dulwich 自动提交知识文件变更，支持追溯和回滚。

### 会话历史（短期记忆）

Session JSONL：每条会话一个文件，完整记录每轮对话（含工具调用），是 Consoldiator 的直接操作对象。

### 完整数据流

```
用户消息 → ContextBuilder 组装上下文（注入三层记忆）
    → Consolidator 检查 token 是否超限，超了就归档
    → LLM 调用 + 工具执行
    → 保存 turn 到 Session JSONL
    →（后台）Dream 定时消费 history.jsonl 更新知识文件
```

### 面试怎么讲

> "内存分两层：知识文件层和事件时间线层。知识文件（SOUL/USER/MEMORY.md）全量注入 System Prompt，由 Dream 两阶段处理器定时消费历史、增量编辑——Phase 1 纯 LLM 分析变更指令，Phase 2 配备工具精确编辑。事件时间线（history.jsonl）JSONL 仅追加，通过自增游标实现增量消费，Agent 用 grep 按需检索，不浪费 token。全体系用 Git 做版本控制，可追溯可回滚。"

## 三、基于 Token 预算的会话压缩

### 要解决什么问题

对话太长 token 会超出模型窗口上限，需要对旧对话做压缩归档，但压缩要聪明——不能随便截断（会割裂 tool_call 链），也不能频繁触发（浪费 LLM 调用）。

### 触发条件

[agent/memory.py](agent/memory.py#L535) Consolidator 的 `_input_token_budget`：

```
预算 = 模型上下文窗口 - 最大回复 token - 安全缓冲
```

每次 LLM 调用前，`estimate_session_prompt_tokens()` 估算当前 prompt 真实 token 数，超过预算就触发压缩。

### 为什么用滞后阈值

目标不是压到预算以下，而是压到 `预算 × 0.5`。如果刚好压到预算线下就停，下一次小幅增长又会立刻触发压缩，反复折腾。降到 50% 给后续对话留出增长空间。

### 边界对齐截断

[agent/memory.py](agent/memory.py#L490) `pick_consolidation_boundary()`：

沿消息列表从 `last_consolidated` 位置向后扫描，**只在 user 角色处设置截断点**。这样保证截断后的历史从一个完整的用户回合开始，不会切断 assistant + tool_call 的中间状态。

### 归档流程

1. LLM 摘要：把待归档的消息块发给 LLM（无工具约束），让它提取关键事实（用户信息/决策/偏好/事件）
2. 摘要写入 history.jsonl
3. 推进 `session.last_consolidated` 游标（即使这条消息不会再被重复处理）
4. 重新估算 token，决定是否继续
5. 最多 5 轮

### 容错设计

LLM 调用失败时，自动降级为 Raw Archive：直接把原始对话内容格式化标记 `[RAW]` 写入 history.jsonl。同时推进游标，避免下一轮重复归档同一段消息（不会 hammer 已降级的 LLM）。

### 面试怎么讲

> "我设计了一个基于 token 预算的压缩器，每次 LLM 调用前估算 prompt token 数，超阈值时自动触发。关键设计有三点：一是沿用户消息边界对齐截断，不割裂 tool_call 链；二是引入滞后阈值，降到预算 50% 才停，避免频繁触发；三是容错兜底，LLM 摘要失败时自动降级为原始归档。整个流程最多 5 轮。"

## 四、RAG 知识库集成

### 要解决什么问题

LLM 的知识受限于训练数据，无法访问用户私有知识（如个人笔记、项目文档）。需要通过外部知识库检索补充上下文。

### IMA 知识库接入方案

IMA（腾讯智能工作台）提供个人知识库能力。通过 MCP 协议将 IMA 知识库暴露为工具，将文档检索包装为标准的 Tool 调用。

### MCP 工具接入机制

入口在 [agent/tools/mcp.py](agent/tools/mcp.py)：

**三种传输方式**：

- **stdio**：启动子进程，通过标准输入输出通信
- **SSE**：Server-Sent Events over HTTP
- **streamableHttp**：流式 HTTP

**连接流程**（懒加载，首次 `run()` 时才连接）：

1. 建立传输连接（stdio 启动子进程 / HTTP 建连）
2. 初始化 MCP ClientSession
3. 获取工具列表 → 包装为 `MCPToolWrapper` → 注册到 `ToolRegistry`
4. 获取资源列表 → 包装为 `MCPResourceWrapper`（只读）
5. 获取提示词列表 → 包装为 `MCPPromptWrapper`（只读）

**MCPToolWrapper**：将外部 MCP 工具包装成标准 Tool 对象：

- 名称格式：`mcp_<server>_<tool>`
- 参数 Schema 自动转换为 OpenAI 兼容格式
- 执行时通过 `session.call_tool()` 远程调用
- 错误的自动重试：瞬时连接错误等 1 秒重试一次

### Skills 机制

[agent/skills.py] 的 `SkillsLoader`：

- 从 `workspace/skills/` 和内置目录发现技能
- YAML frontmatter 元数据（名称、描述、要求、always 标志）
- 渐进加载：系统提示中放摘要，按需读取完整内容
- IMA 知识库技能作为 always 技能注入，在每次对话中可用

### 面试怎么讲

> "基于 MCP 协议接入 IMA 个人知识库，把知识检索包装成标准的只读工具。MCP 工具通过 wrapper 统一注册到工具链，支持 stdio / HTTP 三种传输方式，工具执行时含超时和自动重试。配合 Skills 机制实现渐进加载，知识库技能始终可用但不撑爆上下文。"

## 五、工具统一注册与集成

### 要解决什么问题

- 工具来源多样（本地内置工具、外部 MCP 工具），需要一个统一的注册和执行机制
- LLM 工具调用可能一次返回多个，需要并发执行提高效率
- 有副作用的工具不能并发，需要安全的并发控制

### Tool 抽象基类

[agent/tools/base.py](agent/tools/base.py#L117)：

```python
class Tool(ABC):
    name         # 工具名
    description  # 工具描述（给 LLM 看的）
    parameters   # 参数 JSON Schema
    read_only    # 是否无副作用 → 决定能否并发执行
    exclusive    # 是否需要独占执行
    concurrency_safe  # = read_only and not exclusive
```

关键设计：`read_only` 和 `exclusive` 精细控制并发安全，读写工具不会跟其他工具同时跑。

### ToolRegistry 注册表

[agent/tools/registry.py](agent/tools/registry.py#L8)：

- `register()` / `unregister()` 管理工具
- `get_definitions()` 稳定排序：内置工具在前、MCP 工具在后，顺序固定有利于 LLM 的 Prompt Cache 命中
- `prepare_call()` 参数查找 → 类型转换（字符串转 int/bool 等）→ JSON Schema 校验
- `execute()` 错误处理 + 生成友好的错误提示

### 内置工具注册

[agent/loop.py](agent/loop.py#L345) `_register_default_tools()` 在启动时显式注册：

- 文件：read_file / write_file / edit_file / list_dir
- 搜索：glob / grep
- 网络：web_search / web_fetch
- 其他：exec / ask_user / message / spawn / cron / notebook_edit
- 按配置条件注册（exec 需 enable、web 需 enable）

### MCP 工具注册

懒加载——首次 `run()` 或 `process_direct()` 时才连接 MCP 服务器，避免阻塞启动。连接失败下次消息时会重试。支持 `enabled_tools` 白名单控制暴露哪些工具给 LLM。

### 并发执行控制

[agent/runner.py](agent/runner.py#L1094) `_partition_tool_batches()`：

按 `concurrency_safe` 属性把工具调用分区：

- 只读工具（read_file、glob）→ 同批并行
- 副作用工具（exec、write_file）→ 单独串行

例如 LLM 请求 `[read_file, glob, exec, write_file]`：
→ `[[read_file, glob], [exec], [write_file]]`

### 面试怎么讲

> "基于抽象基类 Tool 做插件式注册，内置工具和 MCP 工具都统一包装实现同一个接口。关键设计是并发安全控制——read_only 属性标记无副作用工具自动并行，exclusive 属性标记独占工具串行执行，一次调用的多个工具按安全属性分区调度，既保证正确性又提高效率。"

## 六、QQ 机器人接入（双队列消息架构）

### 要解决什么问题

智能体核心和通讯平台是两类完全不同的逻辑，需要解耦。而且一个智能体要同时服务多个平台、多个对话，需要异步非阻塞的消息分发。

### MessageBus 双队列设计

[bus/queue.py](bus/queue.py#L8)：

```
MessageBus
├── inbound:  asyncio.Queue   ← 通道把用户消息往里塞
└── outbound: asyncio.Queue   ← 智能体把回复往里塞
```

两条队列各自独立，完全解耦：

- **入站**：`publish_inbound(msg)` — 通道发布 → `consume_inbound()` — AgentLoop 消费
- **出站**：`publish_outbound(msg)` — AgentLoop 发布 → `consume_outbound()` — ChannelManager 分发

### 数据流

```
QQ 消息 → QQBotChannel._handle_message()
    → MessageBus.inbound (put)
    → AgentLoop.run() 轮询消费 → 构建上下文 → LLM + 工具 → 生成回复
    → MessageBus.outbound (put)
    → ChannelManager._dispatch_outbound() 轮询消费
    → QQBotChannel.send() → QQ 回复
```

### BaseChannel 抽象

[channels/base.py]：

所有通道继承 `BaseChannel`，最少实现3个方法：

- `start()` — 启动通道（建立长连接 / 启动 webhook）
- `stop()` — 停止通道
- `send()` — 发送消息

```python
from abc import ABC, abstractmethod

class BaseChannel(ABC):
    """聊天平台的抽象基类——对应 nanobot/channels/base.py"""

    name: str = "base"

    def __init__(self, bus: MessageBus):
        self.bus = bus

    @abstractmethod
    async def start(self):
        """连接平台，开始监听消息"""
        ...

    @abstractmethod
    async def stop(self):
        """断开连接"""
        ...

    @abstractmethod
    async def send(self, msg: OutboundMessage):
        """发送消息到平台"""
        ...

    async def handle_message(self, sender_id: str, chat_id: str, content: str):
        """收到消息时调用——转发到 Bus"""
        await self.bus.publish_inbound(InboundMessage(
            channel=self.name, sender_id=sender_id,
            chat_id=chat_id, content=content,
        ))
```

### ChannelManager 管理

[channels/manager.py](channels/manager.py#L40)：

- `_init_channels()` — 自动发现所有通道插件，按配置过滤启用
- `start_all()` — 并发启动所有通道 + 出站分发器（`asyncio.create_task`）
- `_dispatch_outbound()` — 轮询出站队列，找到对应通道，发送消息
- 流式消息合并优化：连续的 `_stream_delta` 消息合并成一次通道调用，减少 API 请求
- 重试机制：指数退避 1s / 2s / 4s

### QQ 通道

[channels/qq.py]：

- 入站：解析 botpy SDK 消息（C2C 私聊 / Group 群聊），下载附件到本地，调用 `_handle_message()` 发布到总线
- 出站：先发附件（QQ 富媒体 API），再发文本（支持 Markdown），处理了 botpy 版本间附件字段差异

### 面试怎么讲

> "我设计了一个双队列异步消息总线，inbound 队列收、outbound 队列发，把通信层和智能体核心完全解耦。通道层基于 BaseChannel 抽象，每个平台实现 start/stop/send 三个方法就能接入。QQ 机器人就是基于这个架构接入的，消息从收到到回复走完一个完整的异步管道。关键是这个架构让新增平台非常方便，不用改智能体核心代码。"
